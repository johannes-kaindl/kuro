# Kuro CSS Conventions

The rules every edit to `Kuro/src/*.css` follows. They exist so the theme stays clean,
forkable, and friendly to user snippets — and so a future agent can't quietly reintroduce
smelly CSS. Companion docs: **`THEME-AUTHORING.md`** (the [armature]/[values] fork model) and
the design spec **`docs/superpowers/specs/2026-06-23-variable-first-hardening-design.md`**.

`src/check.sh` **enforces** R2, R3, R4 (the build fails otherwise). R1, R5, R6 are review
discipline. Never edit `theme.css` — it is generated by `src/build.sh`.

## The rules

### R1 — Variable-first
If Obsidian exposes a CSS variable for something, **set the variable** (in `12-bridge.css`,
the one place Obsidian-native vars are fed) rather than writing a property override on
Obsidian's elements. Obsidian's own rule then consumes your value with no selector war.
This is the official guidance — *"By overriding the default values for the Obsidian CSS
variables, you can create beautiful themes without the need for complex CSS selectors."*
(Obsidian › About styling). A recurring need to override a property is a smell: find the
variable first (see the map below).

### R2 — `!important` only with justification (enforced)
Every real `!important` declaration must carry an **inline** tag on the same line:

```css
font-family: var(--kuro-font-mono) !important; /* important: Obsidian sets font app-level; CSS vars don't reach it */
```

…**or** live in `src/70-reduced-motion.css` (the accessibility whitelist — its whole purpose
is the `prefers-reduced-motion` blanket reset). `check.sh` strips comments, counts the real
declarations, fails over **9**, and fails on any untagged `!important` outside the whitelist.
Official stance: *"Declaring styles as `!important` prevents users from overriding styles
from your theme using snippets"* (Obsidian › Theme guidelines); *"Don't use `!important`"*
(Obsidian October checklist). Before reaching for it, try in order: **(a)** set the Obsidian
variable (R1); **(b)** match/beat the selector's specificity (themes load after `app.css`, so
an equal-specificity rule wins on source order); **(c)** lower a too-specific *own* base rule
with `:where()` so overrides win naturally (how the checkbox + callout `!important` were
removed). Only if all three fail: tag and justify.

### R3 — Both-mode (enforced)
Every semantic token has a value in **both** `10-semantic-dark.css` and `11-semantic-light.css`,
mirrored key-for-key. `check.sh` fails if the two key sets differ. This is what makes the
theme correct in light *and* dark by construction. (See `THEME-AUTHORING.md` § token flow.)

### R4 — Layer discipline (enforced)
`[armature]` files carry **no** colour literal (`#rgb`) and **no** `--signal-*` / `--void-*` /
`--paper-*` reference — they read tokens by name only. `[values]` files (`00-primitives`,
`02-fonts`, `50-presets`, `55-subthemes`) hold the taste. `check.sh` lints this. A fork swaps
only `[values]`.

### R5 — One component, all surfaces
Define each component **once** across Reading view + Live-Preview + editor source — not
Reading-only. The single `.callout` / `input.task-list-item-checkbox` / `a.tag` selector sets
cover every surface. (This is the trap v4 eliminated; don't reintroduce per-surface forks.)

### R6 — Always build + check
`bash src/build.sh && bash src/check.sh` after every change; it must print `ALL CHECKS PASS`.
Edit fragments, never the generated `theme.css`.

## Verified Obsidian variable map (SSOT)

Grounded in the official reference (researched 2026-06-23). The docs publish **no default
values** — rely on the name/semantics contract, supply your own values for both modes (R3).
When a fix is needed, look here first (R1) before writing any property override.

| Surface | Variables | Source |
|---|---|---|
| **Tabs** | `--tab-background-active`, `--tab-container-background`, `--tab-text-color[-active/-focused/-focused-active/-focused-active-current/-focused-highlighted]`, `--tab-outline-color`, `--tab-outline-width`, `--tab-divider-color`, `--tab-curve`, `--tab-radius`, `--tab-radius-active`, `--tab-font-size`, `--tab-font-weight`, `--tab-stacked-*` | [Components/Tabs](https://docs.obsidian.md/Reference/CSS+variables/Components/Tabs) |
| **Window frame** | `--titlebar-background[-focused]`, `--titlebar-border-width`, `--titlebar-border-color`, `--titlebar-text-color[-focused]`, `--titlebar-text-weight`, `--header-height` | [Window/Window+frame](https://docs.obsidian.md/Reference/CSS+variables/Window/Window+frame) |
| **Checkbox** | `--checkbox-color`, `--checkbox-color-hover`, `--checkbox-border-color`, `--checkbox-border-color-hover`, `--checkbox-marker-color`, `--checkbox-radius`, `--checkbox-size` | [Components/Checkbox](https://docs.obsidian.md/Reference/CSS+variables/Components/Checkbox) |
| **Tag** | `--tag-color[-hover]`, `--tag-background[-hover]`, `--tag-border-color[-hover]`, `--tag-border-width`, `--tag-decoration[-hover]`, `--tag-radius`, `--tag-padding-x`, `--tag-padding-y`, `--tag-size`, `--tag-weight` | [Editor/Tag](https://docs.obsidian.md/Reference/CSS+variables/Editor/Tag) |
| **Callout** | `--callout-radius`, `--callout-padding`, `--callout-border-width`, `--callout-border-opacity`, `--callout-blend-mode`, `--callout-title-*`, `--callout-content-*`, per-type `--callout-<type>` | [Editor/Callout](https://docs.obsidian.md/Reference/CSS+variables/Editor/Callout) |
| **Suggestion / command palette** | **No variable exists** for the selected-item background. Match specificity instead (`.suggestion-item.is-selected` `(0,2,0)`; palette `.prompt.mod-command-palette …` `(0,4,0)`). | — |

**Gotchas:** the Tag page is under `Editor/`, not `Components/` (the latter 404s). There is no
`--titlebar-text-color-highlighted` (the real one is the tab var `--tab-text-color-focused-highlighted`).
The official marker var is `--checkbox-marker-color`, the font-size var is `--tag-size`.
Obsidian does **not** emit `--interactive-accent-rgb` — derive the accent via
`var(--interactive-accent)` + `color-mix()` (see the project's known accent pitfall).
